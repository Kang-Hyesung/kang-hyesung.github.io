## 1. 개요

소프트웨어 수준의 멀티스레딩을 완벽하게 제어하려면 하드웨어의 동작 방식을 이해해야 한다. 스레드의 실행 흐름은 궁극적으로 CPU와 메모리의 상호작용으로 귀결되기 때문이다. 이 글에서는 레지스터와 컨텍스트 스위칭(Context Switching)[^1]의 원리를 짚어보고, JVM(Java Virtual Machine)이 단일 프로세스 환경에서 어떻게 메모리를 다루는지 알아보자.

## 2. 하드웨어 관점의 연산 구조와 상태(State)

CPU가 연산을 수행할 때, 소위 **'현재 상태(State)'**를 정의하는 것은 바로 **레지스터(Register)**의 값이다. 레지스터는 CPU 내부에 존재하는 초고속 메모리로, 직전 혹은 현재 처리 중인 데이터와 명령어의 주소를 담고 있다.

### 2.1 메모리 계층 구조와 연산 흐름

데이터는 주 기억장치(Main Memory)에서 캐시 메모리(L1, L2, L3)를 거쳐 레지스터로 로드된다. CPU는 레지스터에 적재된 값을 바탕으로 연산을 수행하고 그 결과를 다시 레지스터와 메모리에 반영한다.

### 2.2 컨텍스트 스위칭(Context Switching)

운영체제는 여러 스레드를 동시에 실행하는 것처럼 보이게 하기 위해 스레드 간의 실행 흐름을 끊임없이 교체한다.

예를 들어 `T1` 스레드가 `a + b` 연산을 수행하여 결과를 `c`에 저장하려는 찰나에 실행 흐름이 `T2`로 넘어간다고 가정해보자.

1. `T1`의 연산이 중단되면 현재 레지스터에 들어있던 값들(상태)을 어딘가에 저장해야 한다.
2. 이후 운영체제는 `T2`가 이전에 멈췄던 지점의 레지스터 값들을 다시 복원하여 연산을 재개한다.
3. 다시 `T1`의 차례가 돌아오면, 저장해 두었던 `7, 3, 4`와 같은 레지스터 값을 복원하여 `c`에 값을 저장하는 나머지 흐름을 이어간다.

> **Deep Dive: TCB (Thread Control Block)**
> 
> 운영체제는 컨텍스트 스위칭 시 스레드의 상태 정보(프로그램 카운터, 레지스터 값 등)를 TCB(Thread Control Block)라는 자료구조에 저장한다. 스레드가 교체될 때마다 TCB에 상태를 저장하고 읽어오는 오버헤드가 발생하므로, 무분별한 스레드 생성은 오히려 성능 저하를 유발할 수 있다.

## 3. 왜 멀티스레딩을 해야 하는가?

과거에는 CPU의 클럭 주파수나 전압을 높여 연산 속도 자체를 끌어올리는 방식이 주를 이루었다. 하지만 발열과 전력 소모의 물리적 한계에 부딪히면서, 단일 코어의 성능을 높이는 대신 **코어의 개수를 늘리는 멀티 코어 아키텍처**로 패러다임이 전환되었다.

### I/O 병목(Bottleneck)의 극복

시스템에는 CPU 외에도 NIC(네트워크 카드), 사운드 카드 등 다양한 주변 기기가 존재한다. 이러한 I/O 장치들은 CPU에 비해 연산 및 처리 속도가 압도적으로 느리다. 만약 단일 스레드로 작업을 처리한다면, CPU는 네트워크에서 데이터가 도착할 때까지 아무 일도 하지 못하고 대기(Wait)해야 한다. 멀티스레딩은 이러한 **I/O 대기 시간에 다른 스레드의 작업을 처리**하게 함으로써 CPU 자원의 활용도를 극대화한다.

> **Tip:** 파일 다운로드와 게임을 동시에 할 수 있는 이유가 바로 멀티스레딩과 비동기 I/O 처리 덕분이다. I/O 작업 중 블로킹(Blocking)된 스레드를 대신해 다른 스레드가 CPU를 선점하여 게임 로직을 연산한다.
{: .prompt-tip }

## 4. 멀티스레딩 vs 멀티프로세싱

멀티태스킹을 구현하는 방법은 크게 멀티스레딩과 멀티프로세싱으로 나뉜다. 각각은 메모리 관리 관점에서 뚜렷한 장단점을 가진다.

* **멀티스레딩 (Multi-threading)**: 하나의 프로세스 내에서 가상 메모리 공간을 공유한다. 스레드 간 자원 공유가 쉽고 컨텍스트 스위칭 비용이 상대적으로 낮다.
* **멀티프로세싱 (Multi-processing)**: 독립적인 프로세스를 여러 개 실행한다. 프로세스마다 격리된 가상 메모리 공간을 가지므로 데이터 통신을 위해 IPC(Inter-Process Communication)[^2] 기법이 필요하다.

> 멀티스레드 환경에서는 메모리를 공유하므로 단 하나의 스레드에서 발생한 치명적인 오류(예: `OutOfMemoryError`)나 잘못된 메모리 참조가 프로세스 전체를 비정상 종료시킬 수 있다. 안정성이 극도로 중요한 시스템이라면 워커(Worker)를 독립된 프로세스로 분리하는 아키텍처를 고려해야 한다.
{: .prompt-danger }

## 5. JVM 환경에서의 멀티스레딩 (Java Context)

**JVM(Java Virtual Machine)은 OS 관점에서 단일 사용자 모드 어플리케이션 프로세스**로 동작한다. 즉, 우리가 작성하는 일반적인 Java 멀티스레딩 프로그램은 '싱글 프로세스 기반 멀티스레딩' 환경이다.

JVM은 자신에게 할당된 하나의 가상 메모리 공간(Virtual Memory Space)을 사용하여 힙(Heap)과 스택(Stack) 영역을 관리한다. 자바에서 사용하는 모든 **변수**는 결국 이 메모리 공간을 활용하기 위한 문법적 도구다.

### 5.1 Java 구현: 스레드 간 메모리 공유와 동기화 이슈

JVM 환경에서는 스레드들이 Heap 영역을 공유하기 때문에 여러 스레드가 동시에 같은 변수에 접근할 때 가시성 및 동시성 문제가 발생한다.

```java
public class JvmMemoryExample {
    // 힙(Heap) 영역에 저장되어 모든 스레드가 공유하는 자원
    private static int sharedCounter = 0;

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 10000; i++) {
                // 주의: i++는 단일 연산(Atomic)이 아니다. (조회 -> 증가 -> 저장의 3단계)
                // 따라서 동기화 처리 없이 다수의 스레드가 접근하면 경쟁 상태(Race Condition)가 발생한다.
                sharedCounter++; 
            }
        };

        Thread t1 = new Thread(task, "T1");
        Thread t2 = new Thread(task, "T2");

        t1.start();
        t2.start();

        t1.join();
        t2.join();

        // 기대값인 20000보다 작은 값이 출력됨을 확인할 수 있다.
        System.out.println("최종 카운터 값: " + sharedCounter);
    }
}
```

위 코드는 다수의 스레드가 공유 자원(`sharedCounter`)에 접근할 때 왜 동기화(`synchronized`, `AtomicInteger` 등)와 **자바 메모리 모델(JMM)**에 대한 이해가 필요한지 보여주는 대표적인 사례다. 변수는 단순한 값을 넘어, 가상 메모리 공간 위에서 여러 스레드가 어떻게 상호작용하는지를 결정짓는 핵심 요소다.
