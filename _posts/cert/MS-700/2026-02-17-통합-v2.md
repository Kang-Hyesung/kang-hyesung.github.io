## 1) 네트워크 계획/권한/리포팅 (Network Planner / CQD / Call analytics)

### 1-1. Network Planner가 하는 일 = “대역폭 산정 + 보고서”

* **Network Planner**는 지점/사용자 수/WAN 등을 넣어서 **필요 대역폭 계산 + 결과 리포트 생성**에 쓰는 도구.
* 위치: **Teams admin center > Network planner** (경로는 UI 변동 가능하지만 “Teams admin center 안”이 핵심)

**시험에서 답 고르는 기준**

* “지점 배포 전”, “대역폭 요구사항 계산”, “리포트 생성” → **Network Planner**가 정답 후보 1순위

### 1-2. Network Planner 사용 가능한 최소 권한 역할

* Network Planner 실행 가능한 역할: **Global admin / Teams admin / Teams communications admin**
* 최소 권한 원칙(Least privilege) 요구 시 → **Teams communications admin** 선택 ([Microsoft Learn][1])

> ✅ **Support 계열(Role: Teams Communications Support Specialist 등)**은 “설계/계획” 도구 권한이 아니라 “품질/문제 해결(읽기/분석)” 쪽이라서 Network Planner 같은 “계획 도구” 최소권한 정답에서 밀리는 패턴이 자주 나옴.

---

### 1-3. CQD에서 “내부/외부(Inside/Outside)”가 틀리게 찍힐 때

* CQD의 **Inside/Outside(InsideCorp)** 분류는 네트워크 정책(Trusted IP, Network site 등)로 자동 해결되는 게 아니라,
* **Building/endpoint(테넌트/빌딩) 데이터 업로드**로 IP/서브넷을 “내부(InsideCorp)”로 매핑해줘야 함. ([Microsoft Learn][2])

**시험에서 답 고르는 기준**

* “CQD에서 내부 통화가 outside로 찍힘 / 내부·외부 보고 정확히” → **Tenant/Building/Endpoint data 업로드**(테넌트 데이터 업로드) 계열이 정답

---

### 1-4. Call Analytics(통화 분석)에서 “위치(지역) 식별” 강화

여기서 함정이 있었음.

* 예전/일부 문제(연습평가 포함)에서 “Reporting labels 업로드”라고 표현하는데,
  **Reporting labels는 2025-09-15부터 지원 종료(Deprecated)**로 안내됨. ([Microsoft Learn][3])
* 최신 문서 흐름은 **Building and endpoint data로 관리**로 넘어감. ([Microsoft Learn][3])

**시험에서 답 고르는 기준**

* 문제 선지에 **Reporting labels**가 직접 나오면 → 문제 제작 시점/문구를 따라 **그걸 고르게 만드는 문제**가 아직 남아있을 수 있음(너가 본 연습평가가 그 케이스).
* 다만 “최신 기준”으로는 → **Building/endpoint data 업로드로 대체**된다는 걸 머리속에 같이 들고 있어야 함. ([Microsoft Learn][3])

## 2) VDI/Citrix/Azure Virtual Desktop에서 Teams (지원/비지원 포인트)

### 2-1. VDI(WebRTC 최적화)에서 “안 되는 기능”은 따로 정리돼 있다

Microsoft 공식 문서에 **WebRTC 최적화(VDI)**에서 지원 안 되는 항목들이 명시돼 있음. 대표적으로:

* QoS 미지원
* 1080p 미지원
* 사용자가 업로드한 배경 미지원
* **Teams Premium 기능(E2EE/워터마크 등)** 미지원
* Avatars 미지원
* 일부 갤러리 레이아웃/노이즈 억제/줌/Location-based routing/Media bypass 등 제한 ([Microsoft Learn][4])

**시험에서 답 고르는 기준**

* “VDI 환경에서 기능 제한/미지원 무엇?” → **VDI 문서의 ‘WebRTC 최적화에서 지원 안 함’ 리스트에 있는지**로 고름 ([Microsoft Learn][4])
* 특히 **Teams Premium(워터마크/E2EE)** 같은 건 VDI 제약으로 자주 엮임 ([Microsoft Learn][4])

---

### 2-2. Multitenant Multi-Account(MTMA) = VDI에서 ‘완전 미지원’이라고 고르면 위험

* 최신 VDI 문서에는 **“Multitenant and Multi-account in VDI”**가 별도 섹션으로 존재하고,
* 같은 Microsoft 365 클라우드(예: Commercial 내부) 범위에서 계정/조직 전환이 가능하다고 안내됨.
  단, **Cross-cloud guest/meeting 같은 일부는 미지원** ([Microsoft Learn][4])

**시험에서 답 고르는 기준**

* MTMA를 “VDI라서 무조건 안 됨”으로 단정하는 선지는 **최신 문서 기준 위험**

---

### 2-3. Citrix에서 “screen sharing from chat” 관련 이슈 포인트

* VDI 문서에서 **“Screen sharing from chat for Citrix”**는 특정 Citrix Workspace app 버전 조건이 언급되는 식으로 **이슈/제약 포인트로 등장**함 ([Microsoft Learn][4])

**시험에서 답 고르는 기준**

* “VDI/Citrix + screen sharing from chat” 같이 **딱 특정 기능을 찌르는 선지**는 보통 “제약/조건부”로 출제됨 ([Microsoft Learn][4])

## 3) AVD RDP 속성 문제 (오답 많이 나는 포인트)

### 3-1. `audiomode` 값 = “오디오 재생 위치”

* `audiomode:i:<value>`

  * **0 = 로컬 디바이스에서 재생**
  * 1 = 원격 세션에서 재생
  * 2 = 재생 안 함 ([Microsoft Learn][5])

✅ 따라서 “Teams 미디어 최적화 없이, 회의 오디오를 로컬로 보내라(redirect)” 류 문제에서

* **정답 값은 `audiomode:i:0`**가 맞는 방향임 ([Microsoft Learn][5])

### 3-2. RDP 속성을 “어디에 적용하나?” = Host pool이 정답 패턴

* AVD에서 RDP properties는 보통 **Host pool(풀)** 단위에서 설정하는 흐름이 정석(“Host1” 같은 단일 VM이 아니라). ([Microsoft Learn][6])

**시험에서 답 고르는 기준**

* 드롭다운에 **Pool/Host/AppGroup** 같이 나오면
  “RDP property 설정” 문제는 보통 **Pool(Host pool)** 쪽을 고르게 설계됨 ([Microsoft Learn][6])

## 4) 모임/이벤트: Breakout rooms / Copilot / E2EE

### 4-1. Breakout rooms(소회의실) = “일반 Teams 모임(Meetings)에서 참가자 쪼개기”

* Breakout rooms는 **모임 참가자를 여러 소그룹 방으로 나눠 운영**하는 기능.
* “500명 + 콘텐츠 공유/채팅 + breakout rooms” → **Meetings**로 가는 게 일반적인 출제 의도(라이브 이벤트/타운홀은 상호작용 제약이 엮이기 쉬움)

**시험에서 답 고르는 기준**

* “breakout rooms”라는 단어가 나오면 → **Meeting(모임)**이 정답축

---

### 4-2. Copilot을 “회의 중 + 회의 후”까지 쓰려면 먼저 켜야 하는 것

* Copilot 관련 정책은 **Meeting policy > Recording & transcription 영역**에 있음.
* Copilot 옵션 중 **‘saved transcript required’(저장된 전사 요구)** 같은 표현이 공식 가이드에 존재. ([Microsoft Learn][7])
  → 즉 “회의 후까지”를 보장하려면 **전사(Transcription)**이 핵심 전제 조건으로 엮임.

**시험에서 답 고르는 기준**

* “Copilot을 회의 중/후 모두” + 선지에 Transcription이 있으면 → **Transcription 먼저**가 정답 축

---

### 4-3. E2EE(종단 간 암호화)를 켜면 빠지는 기능 (여기서 함정 많음)

E2EE를 켠 회의/통화에서는 기능이 크게 제한됨.
공식 문서의 “사용 불가 기능” 예시:

* **Copilot 사용 불가**
* **Meeting chat 불가**
* 녹화/전사/라이브 캡션 등도 제한

**시험에서 답 고르는 기준**

* “E2EE enabled”가 조건에 있으면 → **Copilot / 채팅 / 녹화/전사/캡션**이 막히는 방향으로 생각
* 보기 중 “Copilot”이 있으면 정답 후보 1순위

## 5) Teams Rooms (Windows) 구성 파일 문제

### 5-1. Teams Rooms on Windows의 대표 설정 파일 = `SkypeSettings.xml`

* “기본 레이아웃/기본 동작 커스터마이즈” 류 문제에서
  Teams Rooms on Windows는 **`SkypeSettings.xml`**을 수정하는 패턴이 핵심 ([Microsoft Learn][8])

**시험에서 답 고르는 기준**

* “Teams Rooms for Windows” + “파일 수정” → **SkypeSettings.xml**부터 찍고 시작 ([Microsoft Learn][8])
  (반대로 JSON 계열은 Teams Rooms on Android/다른 제품과 섞는 함정으로 자주 나옴)

## 6) Teams 앱/확장성(Extensibility): Tab vs Connector vs Message extension

### 6-1. “채널 안에서 앱을 직접 띄워 협업” = **Tab**

* 요구사항: “Project1 채널에서 App1을 직접 열어 사용” + “멤버 협업”
* 이런 패턴은 **Tab(탭)**이 정답 축
  (채널 상단에 앱 화면을 고정해서 공동 작업)

### 6-2. “외부 웹서비스 알림/메시지를 Teams 채널로 받기” = **Connector**

* 외부 시스템 → Teams 채널로 카드/알림 푸시

### 6-3. “게시물/콘텐츠 기반으로 작업(워크플로우) 시작” = **Message extension/Power Automate 계열로 출제**

* 메시지/게시물 컨텍스트에서 액션 실행(검색/트리거)으로 묶어서 출제됨

**시험에서 답 고르는 기준 (한 줄)**

* **화면을 채널에 박아 넣기 = Tab**
* **외부 시스템 알림을 채널로 받기 = Connector**
* **메시지 컨텍스트에서 액션/워크플로우 = Message extension**

## 7) 감사/감사로그: “시간 범위 지정 + 앱 설치/제거”는 어디서?

### 7-1. Teams 앱 설치/제거, 앱 카탈로그 추가/제거 “시간 범위로 추적”

* “특정 날짜/시간 범위” + “누가/언제 했다” → **(통상) Purview Audit(통합 감사 로그)**로 가는 문제 패턴
* Usage report는 “통계/사용량”이지 “감사 이벤트(누가 설치/제거)” 타임라인 증빙이 목적일 때는 부적합

**시험에서 답 고르는 기준**

* “date/time range” + “installed/removed” + “audit” 뉘앙스면 → **Audit log** 계열이 정답

## 8) 새 Teams 배포/설치: VDI에서 대역폭 최소화

### 8-1. `teamsbootstrapper.exe`로 새 Teams 배포 시 “대역폭 최소화” 키워드

* 대역폭 최소화/오프라인/내부 배포 느낌이면:

  * **MSIX(오프라인 패키지)**를 두고 bootstrapper에서 참조하는 흐름으로 출제되는 경우가 많음
  * 공식 문서에 **local MSIX 사용(offline)** 방식이 안내됨

**시험에서 답 고르는 기준**

* “VDI 세션 호스트 여러 대 + 설치 트래픽 최소화” → **MSIX package** 쪽이 정답 축

## 9) 콜 큐/자동 콜백(Callback)

### 9-1. “10분 이상 대기 시 자동 콜백” = Call Queue 기능 영역

* Auto attendant(자동응답) / Dial plan / Route가 아니라
  **대기열(Queue)에서 대기/보류/콜백 같은 UX가 설정되는 패턴**으로 출제됨.

**시험에서 답 고르는 기준**

* “hold/대기열/콜백/대기시간” 키워드 → **Call queue**가 정답축

# 마지막으로: 너한테 특히 중요했던 ‘함정 Top 5’

1. **Network Planner = 대역폭 계산/리포트** + 최소권한은 **Teams Communications Admin** ([Microsoft Learn][1])
2. **CQD Inside/Outside**는 보통 **Tenant/Building/Endpoint data 업로드**로 해결 ([Microsoft Learn][2])
3. **Reporting labels는 현재(문서상) 지원 종료** → Building/endpoint data로 이동 ([Microsoft Learn][3])
4. **E2EE 켜면 Copilot/채팅 등 빠짐** → 보기에서 Copilot 나오면 강력 후보
5. **AVD RDP audiomode**: `audiomode:i:0` = 로컬 재생(리디렉션 방향) ([Microsoft Learn][5]), 적용은 보통 **Host pool** ([Microsoft Learn][6])
